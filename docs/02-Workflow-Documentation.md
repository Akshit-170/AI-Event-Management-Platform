# Workflow Documentation

## AI-Powered Event Management Platform

**n8n Capstone Project – Assignment 7**

This document explains the implementation of the five workflows used in the AI-Powered Event Management Platform. Each workflow is designed to automate a specific stage of the event lifecycle while sharing data through a centralized Google Sheets database.

---

# Project Overview

The platform consists of **five independent n8n workflows** connected through a common database. Instead of directly triggering one another, workflows exchange information through shared Google Sheets tables, making the system modular, easier to maintain, and simple to extend.

The platform automates:

- Participant registration
- QR-based attendance
- Reminder notifications
- Certificate generation
- AI-powered feedback analysis

# Workflow 1 – Registration & Participant Management

## Objective

This workflow automates participant registration by validating submitted data, preventing duplicate registrations, generating a unique QR code, storing participant information, and sending a confirmation email.

**Trigger**

Google Sheets Trigger

A new workflow execution begins whenever a new registration is added to the Registration worksheet.

---

## Workflow Steps

| Step | Node | Purpose |
|------|------|---------|
| 1 | Registration Trigger | Detects a new participant registration |
| 2 | Normalize Data | Formats participant information and creates a unique participant ID |
| 3 | Duplicate Check | Searches existing participant records |
| 4 | Decision | Determines whether the participant already exists |
| 5 | Save Participant | Adds valid registrations to the Participants sheet |
| 6 | Generate QR Code | Creates a unique QR code for attendance |
| 7 | Upload QR | Stores the QR code in Google Drive |
| 8 | Send Email | Sends a confirmation email with the QR code attached |

---

## Workflow Logic

The workflow begins when a participant submits the registration form.

The participant's email is normalized to ensure consistent data storage. A unique participant ID is generated by combining the email address and event ID.

Before creating a new participant record, the workflow checks whether the participant has already registered. Duplicate registrations are ignored, preventing multiple records from being created.

For valid registrations, the participant record is stored in the database. A QR code containing the participant identifier is generated and uploaded to Google Drive.

Finally, a confirmation email containing the QR code is automatically delivered to the participant.

---

## Input

- Registration Form
- Participant Details

## Output

- New participant record
- QR code
- Confirmation email
- Updated Google Drive storage

---

## Services Used

- Google Sheets
- Google Drive
- Gmail
- QR Code API

# Workflow 2 – QR Attendance System

## Objective

This workflow records participant attendance using QR codes while preventing duplicate check-ins and maintaining an audit trail.

**Trigger**

Webhook

The webhook is called whenever a participant's QR code is scanned.

---

## Workflow Steps

| Step | Node | Purpose |
|------|------|---------|
| 1 | QR Webhook | Receives QR scan request |
| 2 | Lookup Participant | Searches participant database |
| 3 | Validate Participant | Checks whether the participant exists |
| 4 | Invalid Scan Handler | Logs invalid QR scans |
| 5 | Duplicate Check | Verifies attendance status |
| 6 | Update Attendance | Records attendance timestamp |
| 7 | Response | Returns confirmation to the scanner |

---

## Workflow Logic

When a participant scans their QR code, the webhook receives the participant ID and searches the Participants database.

If the participant does not exist, the scan is recorded in the Audit Log and an error response is returned.

If the participant has already checked in, the workflow prevents duplicate attendance and informs the user.

For valid first-time check-ins, the workflow updates the attendance status, records the check-in time, and returns a successful confirmation.

---

## Input

- QR Code
- Participant ID

## Output

- Attendance record
- Check-in timestamp
- Audit log (if required)
- Webhook response

---

## Services Used

- Webhook
- Google Sheets
- Respond to Webhook

# Workflow 3 – Event Reminder Automation

## Objective

This workflow automatically reminds participants about their upcoming sessions. It ensures reminders are sent only once and only to participants whose sessions are approaching.

**Trigger**

Schedule Trigger

The workflow executes automatically at regular intervals to check for sessions starting within the reminder window.

---

## Workflow Steps

| Step | Node | Purpose |
|------|------|---------|
| 1 | Schedule Trigger | Starts the workflow automatically |
| 2 | Read Sessions | Retrieves upcoming session details |
| 3 | Filter Sessions | Selects sessions beginning within the next 24 hours |
| 4 | Retrieve Participants | Finds registered participants for each session |
| 5 | Loop Participants | Processes participants one at a time |
| 6 | Send Reminder | Emails reminder information |
| 7 | Wait | Prevents API rate limiting |
| 8 | Update Status | Marks reminder as sent |

---

## Workflow Logic

The workflow begins on a scheduled interval.

It reads all available sessions and identifies those that are scheduled within the next 24 hours. For each eligible session, registered participants are retrieved from the Participants worksheet.

Only participants who have not previously received a reminder are processed.

Each participant receives a personalized reminder email containing session information. After the email is successfully delivered, the participant's record is updated so the reminder will not be sent again.

A short wait period is included between email requests to reduce the chance of exceeding service limits.

---

## Input

- Session schedule
- Participant records

## Output

- Reminder emails
- Updated reminder status

---

## Services Used

- Schedule Trigger
- Google Sheets
- Gmail
- Wait Node

# Workflow 4 – Certificate Generation & Distribution

## Objective

This workflow automatically generates personalized participation certificates and delivers them to attendees after the event has concluded.

**Trigger**

Webhook

The organizer manually starts this workflow once the event has officially ended.

---

## Workflow Steps

| Step | Node | Purpose |
|------|------|---------|
| 1 | Event Completion Webhook | Starts certificate generation |
| 2 | Retrieve Attendees | Reads participants who attended |
| 3 | Filter Eligible Participants | Removes participants who already received certificates |
| 4 | Loop Participants | Processes certificates individually |
| 5 | Generate Certificate | Creates personalized certificate using PDFMonkey |
| 6 | Wait | Allows certificate generation to complete |
| 7 | Check Status | Verifies PDF availability |
| 8 | Download Certificate | Retrieves generated PDF |
| 9 | Upload to Drive | Stores certificate in Google Drive |
|10 | Send Certificate | Emails certificate to participant |
|11 | Update Database | Marks certificate as delivered |
|12 | Finish Workflow | Returns completion response |

---

## Workflow Logic

The workflow begins only after the organizer confirms that the event has ended.

Participant records are filtered to include only attendees who have not yet received a certificate.

Each participant is processed individually. The workflow sends participant information to the PDFMonkey API, which creates a personalized certificate.

Since certificate generation is asynchronous, the workflow waits briefly before checking the document status. Once the PDF is ready, it is downloaded, uploaded to Google Drive, and attached to an email sent to the participant.

Finally, the participant record is updated to indicate that the certificate has already been issued, preventing duplicate deliveries.

---

## Input

- Attendee information
- Certificate template
- Event details

## Output

- PDF certificate
- Email with certificate attachment
- Updated participant record
- Certificate stored in Google Drive

---

## Services Used

- Webhook
- Google Sheets
- PDFMonkey API
- Google Drive
- Gmail

---

## Reliability Features

To improve workflow reliability, the certificate generation process includes several safeguards.

- Participants who already received certificates are skipped.
- PDF generation status is checked before downloading.
- Certificates are stored in Google Drive before being emailed.
- Delivery status is recorded in the database to prevent duplicate emails.
- Retry mechanisms can be configured for temporary API failures.

# Workflow 5 – AI Feedback & Analytics

## Objective

This workflow analyzes participant feedback using AI and generates periodic event reports for organizers. It combines real-time sentiment analysis with scheduled analytics to help organizers understand participant satisfaction and identify areas for improvement.

The workflow contains **two independent branches**, each serving a different purpose.

---

## Branch A – Feedback Processing

**Trigger**

Google Sheets Trigger

The workflow starts whenever a new feedback response is submitted through the feedback form.

### Workflow Steps

| Step | Node | Purpose |
|------|------|---------|
| 1 | Feedback Trigger | Detects new feedback submissions |
| 2 | Prepare Feedback | Extracts participant information and comments |
| 3 | AI Analysis | Determines sentiment and identifies the main feedback theme |
| 4 | Parse AI Response | Converts AI output into structured fields |
| 5 | Store Feedback | Saves the analyzed feedback into the database |

---

### Workflow Logic

When a participant submits feedback, the workflow extracts the rating and comments before sending them to the AI model.

The model classifies the overall sentiment (Positive, Neutral, or Negative) and identifies the primary discussion topic or theme.

The analyzed feedback is then stored in the Feedback worksheet, making it immediately available for future reporting.

---

## Input

- Participant feedback
- Rating
- Comments

## Output

- Sentiment classification
- Theme classification
- Updated feedback database

---

## Branch B – Weekly Analytics Report

**Trigger**

Schedule Trigger

Runs automatically once every week.

### Workflow Steps

| Step | Node | Purpose |
|------|------|---------|
| 1 | Weekly Schedule | Starts report generation |
| 2 | Read Participants | Retrieves registration and attendance data |
| 3 | Read Feedback | Retrieves analyzed feedback |
| 4 | Merge Data | Combines participant and feedback information |
| 5 | Prepare Statistics | Calculates event metrics |
| 6 | AI Report Generation | Creates a summary using AI |
| 7 | Send Report | Emails the report to organizers |

---

### Workflow Logic

Every week, the workflow collects participant information together with AI-tagged feedback.

Basic statistics such as attendance count, participant count, and average ratings are calculated before being sent to the AI model.

The AI generates a concise summary highlighting:

- Overall participant satisfaction
- Attendance performance
- Frequently mentioned topics
- Common suggestions
- Recommendations for future events

The completed report is automatically emailed to the event organizers.

---

## Input

- Participant records
- Attendance data
- Feedback database

## Output

- Weekly analytics report
- Organizer email

---

## Services Used

- Google Sheets
- Schedule Trigger
- OpenAI-compatible LLM
- Gmail

            │

            ▼

This modular architecture allows each workflow to be tested, maintained, and extended independently without affecting the others.
